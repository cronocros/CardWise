# cardwise-review-team: 코드 리뷰 전용 팀

새 코드에 대해 Security / Quality / Performance 3개 관점으로 병렬 리뷰를 수행한다.

## 사용 시점

- 기능 구현 완료 후
- PR 생성 전
- 사용자가 "리뷰", "검토", "체크" 등을 언급할 때

## 리뷰 프로세스

변경된 파일 목록을 파악한 뒤, 3개 리뷰 에이전트를 **병렬로** 실행한다.

### Agent 1: Security Review

프롬프트:
```
다음 파일들에 대해 보안 리뷰를 수행해라. CardWise 프로젝트 컨텍스트:
- Supabase PostgreSQL + Auth (JWT)
- Spring Boot (Kotlin) 백엔드
- Next.js 15 프론트엔드

체크리스트:
1. Supabase RLS 정책 누락: 새 테이블에 RLS가 설정되었는가?
2. API 키 노출: ANTHROPIC_API_KEY 등이 클라이언트 번들에 포함되지 않는가?
3. 입력 검증: API 엔드포인트에 Bean Validation이 적용되었는가?
4. SQL Injection: raw SQL 사용 여부, 파라미터 바인딩 확인
5. JWT 검증: 보호 라우트에 인증 미들웨어가 적용되었는가?
6. Rate Limiting: AI/결제 관련 엔드포인트에 레이트 리미팅이 있는가?
7. XSS/CSRF: 사용자 입력이 적절히 이스케이프되는가?
8. 환경변수: .env 파일이 커밋에 포함되지 않는가?

각 항목별 PASS/FAIL/N-A 판정 + 이유를 보고해라.
```

### Agent 2: Quality Review

프롬프트:
```
다음 파일들에 대해 코드 품질 리뷰를 수행해라. CardWise 아키텍처:
- 헥사고날 아키텍처 (Domain -> Port -> Adapter)
- 모듈러 모놀리스 (모듈 간 직접 의존 금지, 이벤트 통신)
- DDD (Aggregate 간 ID 참조, 하나의 트랜잭션 = 하나의 Aggregate)

체크리스트:
1. 헥사고날 구조 준수: Domain이 외부 의존성을 갖지 않는가?
2. 모듈 간 결합: 다른 모듈의 내부 클래스를 직접 참조하지 않는가?
3. 이벤트 사용: 모듈 간 통신이 도메인 이벤트로 이루어지는가?
4. Aggregate 경계: 하나의 트랜잭션에서 여러 Aggregate를 수정하지 않는가?
5. 네이밍: snake_case(DB), camelCase(Kotlin/TS) 준수
6. 에러 처리: 도메인 예외가 적절히 정의되었는가?
7. 테스트: 핵심 로직에 단위 테스트가 있는가?

각 항목별 판정 + 구체적 위치(파일:라인) 보고.
```

### Agent 3: Performance Review

프롬프트:
```
다음 파일들에 대해 성능 리뷰를 수행해라.

체크리스트:
1. N+1 쿼리: JPA 연관 관계에서 N+1 문제가 없는가?
2. 인덱스 활용: 새 쿼리가 기존 인덱스를 활용하는가? 새 인덱스가 필요한가?
3. 캐시: Redis 캐시가 적절히 활용되는가? 캐시 무효화가 빠지지 않았는가?
4. 불필요한 데이터 로드: SELECT * 대신 필요한 컬럼만 조회하는가?
5. 페이지네이션: 목록 조회에 페이지네이션이 적용되었는가?
6. 비동기 처리: 이벤트 핸들러가 @Async로 비동기 처리되는가?
7. 프론트엔드: 불필요한 리렌더링, 큰 번들 사이즈 문제가 없는가?

각 항목별 판정 + 개선 제안 보고.
```

3개 에이전트 결과를 종합하여 사용자에게 보고:

```

### Security: {PASS 수}/{전체}
- [FAIL] {항목}: {이유} ({파일:라인})

### Quality: {PASS 수}/{전체}
- [FAIL] {항목}: {이유} ({파일:라인})

### Performance: {PASS 수}/{전체}
- [FAIL] {항목}: {이유} ({파일:라인})

### 필수 수정 사항 (블로커)
1. {항목}

### 권장 수정 사항
1. {항목}
```
