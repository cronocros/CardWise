## 상위 리스크 상세 대응 계획

### R-B1: 카드사 실적 정책 변경 (최우선)

**현황**: 카드사는 연 1~2회 실적 구간을 변경함. 현재 설계에서 구간값이 소스코드에 하드코딩될 경우, 변경마다 배포 필요.

**대응**:
1. 기존 `performance_tier` 테이블(schema-design.md 기준)로 구간 DB 관리:
   - `card_id`, `tier_name`, `min_amount`, `max_amount` 컬럼으로 구간 정의
   - 소스코드 하드코딩 금지 — 구간값은 반드시 DB 조회
   - 관리자 API로 구간 업데이트 (배포 없이)
2. 카드사 홈페이지 변경 감지 크롤러 (월 1회 체크)
3. 구간 변경 시 영향받는 `user_card_benefit` 캐시 무효화

**모니터링**: 파싱 성공률 < 95% 시 Slack 알림

---

### R-B2: 혜택 정보 부정확

**현황**: 크롤링 기반 혜택 데이터는 카드사 정책 변경 시 즉시 반영되지 않을 수 있음.

**대응**:
1. 모든 혜택 정보에 `last_crawled_at` 표시 (7일 초과 시 "최신 정보 확인 권장" 배지)
2. UI에 "카드사 기준 최종 확인 필수" 면책 문구 상시 표시
3. 사용자 오류 신고 기능 (Phase 1)

---

### R-T3: 이메일 파싱 실패

**현황**: 각 카드사마다 이메일 HTML 구조가 다르고, 변경 시 파싱 실패.

**대응**:
1. 파싱 성공률 모니터링 (카드사별 일간 통계)
2. 실패 시 사용자에게 "수동 입력" 안내 푸시
3. 파싱 실패 건 어드민 대시보드 노출, 담당자 검토 후 규칙 업데이트
4. Fallback: 사용자가 직접 금액만 입력 (간편 수동 입력 모드)
