# F6 - 바우처 관리

> 최종 갱신: 2026-03-19

---

## 1. 개요

### 목적

사용자 카드에 포함된 바우처(쿠폰, 서비스, 라운지, 보험 등 별도 사용/수령이 필요한 혜택)의 사용 현황을 추적하고 관리하는 기능을 정의한다. Benefit(결제 시 자동 적용)과 달리 Voucher는 사용자가 직접 사용 여부를 체크해야 하므로, 잔여 횟수 추적과 만료 알림이 핵심이다.

일부 바우처는 잠금해제 조건(`unlock_conditions` JSONB)을 가지며, 연간실적 달성 등 특정 조건 충족 시 자동 또는 수동으로 잠금이 해제된다.

월간/연간 초기화 주기에 따라 바우처 횟수가 자동 리셋되며, 만료 임박 바우처에 대한 알림을 통해 혜택 누락을 방지한다.

### 대상 사용자

- 카드 부가 서비스(라운지, 발렛, 보험 등)를 적극 활용하는 회원
- 월간 쿠폰이나 할인권을 놓치지 않고 사용하고 싶은 회원
- 여러 카드의 바우처를 한눈에 관리하고 싶은 회원

|--------|------|------|

| 사용 조건 | Text | 실적 구간 요구 등 |
| 잔여/총 횟수 | ProgressBar | 시각적 잔여 표시 |
| 사용 이력 | Timeline | 날짜별 사용/되돌리기 기록 |
| 다음 초기화 | Text | "다음 초기화: 2026-04-01" |
| 잠금해제 조건 | ConditionCard | `unlock_conditions` JSONB 기반 표시 (아래 참조) |
| 사용 처리 버튼 | Button (primary) | "사용 처리" |

**잠금해제 조건 표시 규칙**:
- **잠금 상태**: 🔒 + "연간실적 300만원 달성 시 사용 가능" (조건 미충족)
- **잠금해제 유형**:
  - `AUTO` — 조건 달성 시 시스템이 자동으로 잠금해제
  - `MANUAL` — 조건 달성 후 사용자가 "잠금해제 신청" 버튼 클릭
  - `ADMIN` — 관리자(카드사)가 수동으로 잠금해제
- **잠금해제 후**: 일반 바우처와 동일하게 사용 가능 (사용 처리 버튼 활성화)

## 4. Acceptance Criteria

| # | Given | When | Then |
|---|-------|------|------|
| AC-01 | 바우처 목록 화면에서 | 카드를 선택하면 | 해당 카드의 바우처만 필터링되어 표시된다 |
| AC-02 | "공항 라운지 2/3회" 바우처에서 | "사용 처리"를 누르면 | 잔여 횟수가 1/3으로 변경된다 |
| AC-03 | 사용 처리된 바우처에서 | "되돌리기"를 누르면 | 잔여 횟수가 원복되고 사용 이력에 "되돌리기" 기록이 추가된다 |
| AC-04 | 월간 바우처의 월 초가 되면 | 자동 초기화가 실행되고 | 잔여 횟수가 총 횟수로 리셋된다 |
| AC-05 | 바우처 만료 D-7일에 | 알림이 생성되고 | 사용자에게 만료 임박 알림이 표시된다 |
| AC-06 | 잔여 횟수가 0인 바우처에서 | "사용 처리"를 시도하면 | "잔여 횟수가 없습니다" 에러가 표시된다 |
| AC-07 | 바우처 목록에서 | 만료 임박 바우처가 있으면 | 상단 만료 임박 섹션에 노란색 배경으로 강조 표시된다 |
| AC-08 | 연간실적 조건이 있는 AUTO 유형 바우처에서 | 사용자의 연간실적이 조건 금액을 달성하면 | 바우처가 자동으로 잠금해제되고 사용 가능 상태가 된다 |
| AC-09 | MANUAL 유형 바우처에서 | 조건을 달성한 상태이면 | "잠금해제 신청" 버튼이 활성화되고 클릭 시 잠금해제 요청이 처리된다 |

## 5. Edge Cases & 에러 시나리오

| 시나리오 | 조건 | 처리 |
|----------|------|------|
| 횟수 제한 없는 바우처 | total_count = NULL | "무제한" 표시, ProgressBar 대신 "∞" |
| 동시 사용 처리 | 다중 디바이스에서 동시 사용 처리 | Optimistic Lock (version 컬럼) |
| 카드 삭제 시 | 바우처가 있는 카드를 삭제할 때 | 바우처도 함께 비활성화 (soft delete) |
| 바우처 데이터 갱신 | 카드사 바우처 정보가 변경된 경우 | 크롤러 갱신 시 신규 바우처 추가, 기존 매핑 유지 |
| 이미 만료된 바우처 | 만료일이 지난 바우처 | 회색 처리, "만료됨" 표시, 사용 처리 불가 |
| 실적 구간 미달 | 바우처 사용 조건에 실적 구간이 있는 경우 | "실적 미달" 안내, 사용 처리는 허용 (카드사 정책 차이) |
| 잠금 바우처 조건 미달 시 | unlock_conditions 조건 미충족 상태 | 회색 처리, 남은 금액/조건 표시, 사용 처리 불가 |

## 7. 데이터 모델 연동

### 관련 테이블

| 테이블 | 역할 |
|--------|------|
| `card_voucher` | 카드 바우처 마스터 (카드사 제공 정보) |
| `user_voucher` | 사용자별 바우처 상태 (used_count, reset 주기) |
| `user_card` | 사용자 등록 카드 |
| `card` | 카드 마스터 |
| `notification` | 만료 알림 |
| `card_performance_exclusion` | 실적 제외 항목 (간접 연관 — 잠금해제 조건 판단 시 참조) |

> **참고**: `card_voucher.unlock_conditions` (JSONB) 컬럼에 잠금해제 조건이 정의된다. 조건 유형(연간실적, 전월실적 등), 목표 금액, 잠금해제 방식(AUTO/MANUAL/ADMIN) 등을 구조화하여 저장한다.

### 핵심 비즈니스 로직

1. **사용 처리**: `used_count += 1`, 이력 기록, `total_count` 초과 시 거부
2. **되돌리기**: `used_count -= 1` (0 미만 불가), 이력 기록
3. **월간 초기화**: 매월 1일 배치 → `reset_cycle = 'MONTHLY'`인 바우처 `used_count = 0`
4. **연간 초기화**: 카드 발급일 기준 연간 주기 → `used_count = 0`
5. **만료 알림**: D-7, D-3, D-1에 notification 생성 (NotificationCreatedEvent)
6. **잠금해제 처리**: `card_voucher.unlock_conditions` 확인 → 조건 충족 시 활성화 이벤트 (VoucherUnlockedEvent) 발행 → AUTO 유형은 즉시 잠금해제, MANUAL 유형은 사용자 신청 후 처리
